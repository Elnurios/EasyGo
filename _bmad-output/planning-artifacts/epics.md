# EasyGo - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for EasyGo, decomposing the requirements from the PRD and Architecture into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: Водитель может зарегистрироваться как автопредприниматель (ИП/самозанятый)
FR2: Водитель получает топливный бонус 1 000 ₽ при первой регистрации как автопредприниматель
FR3: Пассажир может зарегистрироваться и создать профиль
FR4: Система может верифицировать номер телефона пользователя
FR5: Администратор может управлять статусом водителя (активен / приостановлен / заблокирован)
FR6: Водитель может принимать заказы через приложение TaxiMaster
FR7: Пассажир может заказать поездку через приложение TaxiMaster
FR8: Диспетчер может распределять заказы через диспетчерскую TaxiMaster
FR9: Система может получать данные о завершённых поездках из TaxiMaster API
FR10: Система может отображать координаты водителя в реальном времени
FR11: Водитель может видеть свою чистую прибыль за любой период
FR12: Водитель может видеть себестоимость каждой поездки (12 ₽/км — своё авто, 17 ₽/км — аренда)
FR13: Водитель может видеть статистику поездок (количество, километраж, часы)
FR14: Водитель может видеть свой рейтинг и историю оценок
FR16: Водитель может видеть начисления по реферальной программе
FR17: Водитель может получить персональную реферальную ссылку
FR18: Пассажир может получить персональную реферальную ссылку
FR19: Система начисляет 2% с поездок реферала водителю-рефереру
FR20: Система начисляет 2% с поездок реферала пассажиру-рефереру
FR21: Водитель и пассажир могут видеть историю реферальных начислений
FR22: Пассажир может оставить оценку водителю после поездки
FR23: Водитель может оставить оценку пассажиру после поездки
FR24: Водитель видит рейтинг пассажира перед принятием заказа
FR25: Водитель может подать апелляцию на спорную оценку
FR26: Система фиксирует историю нарушений и жалоб
FR27: Администратор может видеть дашборд в реальном времени (водители онлайн, заказы, выручка)
FR28: Администратор может просматривать реестр водителей с профилями и документами
FR29: Администратор может управлять статусами водителей и пассажиров
FR30: Администратор может видеть финансовую аналитику платформы (выручка, рефералы, бонусы)
FR31: Администратор может видеть статус интеграции TaxiMaster и лог ошибок
FR33: Администратор может видеть динамику роста водителей и пассажиров
FR41: Водитель может активировать режим параллельной работы при регистрации — профиль отмечается флагом "работает параллельно с другим агрегатором"
FR42: Любой пользователь может видеть публичный финансовый отчёт платформы в реальном времени
FR43: Пассажир, зарегистрировавшийся по реферальному коду водителя, получает скидку 30% на первые 10 поездок
FR44: Скидку 30% покрывает платформа; платформа может предложить водителю добровольно покрыть часть скидки (до 2 поездок) — только с явного согласия водителя
FR45: Водитель видит в личном кабинете предложения платформы об участии в скидке и может принять или отклонить каждое предложение отдельно
FR46: Пассажир может добавить водителя в список избранных после завершённой поездки
FR47: Пассажир видит статус (онлайн / офлайн) каждого избранного водителя в приложении
FR48: Пассажир может отправить прямой запрос на поездку конкретному избранному водителю
FR49: Система уведомляет пассажира, что время подачи может быть увеличено, если избранный водитель находится далеко от точки подачи
FR50: Водитель получает уведомление о прямом запросе от избранного пассажира и самостоятельно принимает решение принять или отклонить заказ
FR51: Поездка с избранным водителем тарифицируется по надбавке +20–30% к базовому тарифу (конкретный процент устанавливается по договорённости с сообществом)
FR52: При отклонении запроса водителем или его недоступности система предлагает пассажиру стандартный заказ без избранного водителя

### NonFunctional Requirements

NFR1: Личный кабинет водителя загружается и отображает актуальные данные менее чем за 3 секунды
NFR2: Мастер-панель основателя обновляет данные с задержкой не более 60 секунд
NFR3: Синхронизация данных поездок из TaxiMaster выполняется каждые 60 секунд с частотой ошибок не более 0,1% операций за сутки
NFR4: Система выдерживает одновременную работу 100 водителей в MVP, до 1 000 в Growth-фазе (измеряется нагрузочным тестированием)
NFR5: Все персональные данные пользователей хранятся на серверах в РФ (ФЗ-152, 242-ФЗ)
NFR6: Все данные шифруются при передаче (HTTPS/TLS) и хранении
NFR7: Доступ к мастер-панели ограничен ролями: только founder / admin
NFR8: Финансовые операции (рефералы, бонусы) логируются с полным аудит-треилом
NFR9: KYC-верификация водителей-ИП/самозанятых перед первой выплатой (ФЗ-115)
NFR10: Платёжные данные обрабатываются только через сертифицированный шлюз (PCI DSS на стороне провайдера)
NFR11: Архитектура поддерживает горизонтальное масштабирование для роста в 10x без переработки
NFR12: Схема базы данных поддерживает добавление до 10 городов в Growth-фазе без изменения архитектуры
NFR13: Реферальная система обрабатывает цепочки до 5 уровней вложенности с временем расчёта не более 5 секунд на транзакцию
NFR14: Доступность платформы — не менее 99% (не более 7 часов простоя в месяц)
NFR15: При недоступности TaxiMaster API система сохраняет последние известные данные и отображает статус ошибки
NFR16: Реферальные начисления и финансовые операции идемпотентны — дублирование исключено

### Additional Requirements

- TaxiMaster API polling каждые 30–60 сек (webhooks отсутствуют — только периодические запросы)
- Аутентификация TaxiMaster: MD5-подпись параметров + секретный ключ, заголовок Signature
- Минимальная версия TaxiMaster: 3.15
- EasyGo Backend: собственная БД для агрегации данных из TaxiMaster
- Расчёт себестоимости по профилю водителя: 12 ₽/км (своё авто) / 17 ₽/км (аренда)
- Настройка TaxiMaster с тарифом 0% комиссии EasyGo
- Собственная система рейтингов и рефералов (не входит в TaxiMaster API)
- Greenfield проект: нужна история настройки проекта и среды разработки (Story 0)
- Хранение данных только на серверах в РФ

### UX Design Requirements

Нет UX-документа. UI подразумевается для двух поверхностей:
- Личный кабинет водителя (веб/мобильный дашборд)
- Мастер-панель администратора (веб-панель)

## Epic List

### Эпик 1: Водитель и пассажир присоединяются к EasyGo

Водитель регистрируется как автопредприниматель, проходит верификацию телефона и получает топливный бонус 1 000 ₽. Пассажир создаёт профиль. Оба понимают концепцию нулевого риска — можно работать параллельно с текущим агрегатором.

**FRs covered:** FR1, FR2, FR3, FR4, FR41

### Эпик 2: TaxiMaster запущен и данные поступают в EasyGo

TaxiMaster настроен с тарифом 0% комиссии EasyGo. Водители выполняют заказы через приложение TaxiMaster. EasyGo Backend синхронизирует данные поездок через API (polling каждые 30–60 сек) и хранит их в собственной БД.

**FRs covered:** FR6, FR7, FR8, FR9, FR10

### Эпик 3: Водитель видит свой бизнес в личном кабинете

Водитель открывает личный кабинет EasyGo и без инструкций понимает: сколько заработал (чистая прибыль), сколько стоила каждая поездка (12 или 17 ₽/км), статистику поездок, свой рейтинг и реферальные начисления.

**FRs covered:** FR11, FR12, FR13, FR14, FR16

### Эпик 4: Реферальная программа и рейтинги работают

Водители и пассажиры получают реферальные ссылки и зарабатывают 2% с поездок приглашённых — деньгами, не баллами. После каждой поездки — взаимные оценки (1–5), которые отображаются в кабинете и повышают доверие сообщества.

**FRs covered:** FR17, FR18, FR19, FR20, FR21, FR22, FR23, FR24, FR25, FR26

### Эпик 5: Администратор управляет платформой EasyGo

Основатель в реальном времени видит дашборд, реестр водителей, финансовую аналитику, статус интеграции TaxiMaster и динамику роста. Может управлять статусами участников. Публичный финансовый отчёт доступен всем.

**FRs covered:** FR5, FR27, FR28, FR29, FR30, FR31, FR33, FR42

### Эпик 6: Пассажирская стратегия — реферальные скидки и Избранный водитель

Водитель приглашает пассажира по персональному коду — пассажир получает 30% скидку на первые 10 поездок, водитель зарабатывает 2% навсегда. Пассажир добавляет понравившегося водителя в Избранные, видит его статус онлайн/офлайн и отправляет прямой запрос на поездку. Система честно предупреждает о времени ожидания; поездка с Избранным стоит дороже (+20–30%), но пассажир выбирает сам.

**FRs covered:** FR43, FR44, FR45, FR46, FR47, FR48, FR49, FR50, FR51, FR52

## Epic 1: Водитель и пассажир присоединяются к EasyGo

Водитель регистрируется как автопредприниматель, проходит верификацию телефона и получает топливный бонус 1 000 ₽. Пассажир создаёт профиль. Оба видят концепцию нулевого риска — работа параллельно с текущим агрегатором.

### Story 1.1: Инициализация проекта EasyGo

Как разработчик,
я хочу развернуть базовый проект EasyGo с БД, системой ролей и API-скелетом,
чтобы все последующие истории имели готовую инфраструктуру.

**Acceptance Criteria:**

**Given** чистый сервер / dev-среда
**When** разработчик выполняет первоначальную настройку
**Then** проект запускается, БД содержит таблицы: users, roles (founder/admin/driver/passenger), миграции применены
**And** базовый JWT auth endpoint доступен
**And** все данные хранятся на сервере в РФ (NFR5)

### Story 1.2: Верификация номера телефона

Как новый пользователь,
я хочу подтвердить свой номер телефона через SMS-код,
чтобы система убедилась в подлинности моего аккаунта.

**Acceptance Criteria:**

**Given** пользователь вводит номер телефона
**When** нажимает "Получить код"
**Then** на номер приходит SMS с 6-значным кодом в течение 30 секунд
**And** код действителен 5 минут
**And** после 3 неверных попыток — блокировка на 15 минут
**And** после успешной верификации номер привязан к аккаунту

### Story 1.3: Регистрация водителя-автопредпринимателя

Как водитель-автопредприниматель,
я хочу зарегистрироваться на EasyGo за 3 минуты,
чтобы начать использовать платформу параллельно с текущим агрегатором без риска.

**Acceptance Criteria:**

**Given** водитель открывает страницу регистрации
**When** заполняет: ФИО, телефон (верификация через 1.2), тип занятости (ИП / самозанятый), тип авто (своё / аренда)
**Then** аккаунт создаётся с ролью driver, профиль сохранён
**And** на экране отображается онбординг-сообщение: "Продолжай работать в Яндексе — возьми 2–3 заказа через EasyGo параллельно. Без комиссии. Посмотри сам."
**And** водитель перенаправляется на экран получения бонуса

### Story 1.4: Приветственный топливный бонус

Как только что зарегистрированный водитель,
я хочу получить топливный бонус 1 000 ₽ при первой регистрации,
чтобы сразу ощутить материальную выгоду от присоединения к EasyGo.

**Acceptance Criteria:**

**Given** водитель завершил регистрацию (История 1.3)
**When** регистрация подтверждена впервые (не повторная)
**Then** в кабинете водителя отображается бонус 1 000 ₽ со статусом "начислен"
**And** бонус начисляется только один раз на один номер телефона
**And** в аудит-логе фиксируется операция начисления бонуса (NFR8)

### Story 1.5: Регистрация пассажира

Как пассажир,
я хочу создать профиль в EasyGo,
чтобы участвовать в реферальной программе и оставлять рейтинги водителям.

**Acceptance Criteria:**

**Given** пользователь открывает страницу регистрации пассажира
**When** вводит имя, телефон (верификация через 1.2) и подтверждает согласие на обработку данных
**Then** аккаунт создаётся с ролью passenger
**And** пользователь получает персональную реферальную ссылку (доступна в профиле)
**And** данные хранятся на серверах в РФ (NFR5, ФЗ-152)

## Epic 2: TaxiMaster запущен и данные поступают в EasyGo

TaxiMaster настроен с тарифом 0% комиссии EasyGo. EasyGo Backend синхронизирует данные поездок через API (polling каждые 30–60 сек) и хранит их в собственной БД — фундамент для кабинета водителя.

### Story 2.1: Настройка TaxiMaster с тарифом 0% EasyGo

Как администратор EasyGo,
я хочу настроить TaxiMaster с тарифом 0% комиссии и подключить API-ключи,
чтобы водители и пассажиры работали через TaxiMaster без комиссии агрегатора.

**Acceptance Criteria:**

**Given** доступ к административной панели TaxiMaster (версия ≥ 3.15)
**When** администратор применяет конфигурацию EasyGo
**Then** тариф EasyGo установлен с комиссией 0%
**And** API-ключ и MD5-секрет сохранены в переменных окружения (не в коде)
**And** приложение водителя, приложение пассажира и диспетчерская TaxiMaster доступны и функционируют
**And** тестовый API-запрос get_driver_info возвращает "code": 0

### Story 2.2: Сервис синхронизации данных поездок из TaxiMaster

Как система EasyGo,
я хочу получать данные о завершённых поездках из TaxiMaster API каждые 30–60 секунд,
чтобы кабинет водителя всегда отображал актуальную статистику.

**Acceptance Criteria:**

**Given** TaxiMaster API доступен и настроен (История 2.1)
**When** EasyGo Backend выполняет polling каждые 30–60 секунд
**Then** новые завершённые поездки из get_completed_orders сохраняются в EasyGo БД (таблица rides)
**And** повторная синхронизация одной поездки не создаёт дубликат (NFR16, идемпотентность)
**And** каждая запись содержит: driver_id, километраж, выручка, время завершения
**Given** TaxiMaster API недоступен
**When** polling выполняется
**Then** EasyGo сохраняет последние известные данные и логирует ошибку (NFR15)

### Story 2.3: Координаты водителя в реальном времени

Как администратор EasyGo,

**Acceptance Criteria:**

**Given** пассажир видит избранного водителя со статусом "онлайн" в разделе "Мои водители"
**When** нажимает "Запросить поездку"
**Then** система отображает предупреждение о возможном увеличении времени ожидания с примерным временем подачи (FR49)
**And** пассажир видит тариф поездки с надбавкой +20–30% к базовому тарифу и явную пометку "Тариф Избранного водителя" (FR51)
**And** пассажир может подтвердить запрос или отменить и вернуться к стандартному заказу

**Given** пассажир подтвердил прямой запрос
**When** запрос отправлен
**Then** водитель получает push-уведомление: имя пассажира, точка подачи, тариф и метка "Запрос от Избранного пассажира" (FR50)
**And** водитель самостоятельно принимает или отклоняет запрос в течение 60 секунд
**And** если водитель принял — поездка оформляется в TaxiMaster по тарифу с надбавкой
**And** начисление 2% реферальных водителю-рефереру (если применимо) рассчитывается от итоговой суммы с надбавкой

**Given** водитель отклонил запрос или не ответил в течение 60 секунд (FR52)
**When** время ожидания истекло или пришёл отказ
**Then** пассажиру отображается уведомление: "Водитель недоступен. Оформить стандартный заказ?"
**And** при согласии — пассажир переходит в стандартный флоу заказа TaxiMaster без надбавки
**And** при отказе — пассажир возвращается в раздел "Мои водители"
**And** отклонённый запрос не влияет на рейтинг ни водителя, ни пассажира
